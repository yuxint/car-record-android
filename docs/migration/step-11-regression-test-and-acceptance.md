# Step 11 验收文档：回归测试矩阵与一致性检查

## 1. 目标与范围
- 目标：建立迁移完成前的质量门禁，验证与 iOS 行为一致性。
- 范围：单测、关键集成测试、契约测试、验收清单。
- 非范围：性能极限压测。

## 2. 前置依赖
- 主约束：`step-01-foundation-tech-decisions.md`
- 上一步：`step-10-ui-shell-and-feature-wiring.md`

## 3. 设计与实现清单
- 测试矩阵
  - 数据约束：`cycleKey`、`cycleItemKey` 唯一性。
  - 规则：提醒计算、排序、颜色分段。
  - 同步：记录保存去重、关系重建、里程联动。
  - JSON：导入导出兼容与异常校验。
  - 上下文：manual now、applied car 回退、导航请求。
- 一致性检查
  - 与 iOS 样本输入输出逐项对照。
  - 列出可接受差异与不可接受差异。

## 4. 验收标准
- 核心规则与数据约束测试通过。
- 双向 JSON 兼容测试通过。
- 完成定义（DoD）
  - 测试矩阵覆盖关键业务路径。
  - 验收清单可执行且结果可追溯。
  - 不可接受差异为 0。

## 5. 风险与回滚
- 风险
  - 样本不足导致“看似通过，实际偏差”。
  - 只做 happy path，遗漏异常路径。
- 回滚策略
  - 以失败用例驱动修复；禁止带已知关键失败进入 Step 12。

## 6. 决策记录
- 候选方案对比
  - 最小烟测：快但风险高。
  - 核心规则+仓储单测（推荐）：投入与收益平衡。
  - 高覆盖含大量集成：质量更稳但周期长。
- 状态
  - 当前采用“核心规则+仓储单测”为默认门禁。
